# Manas オンボーディング設計仕様書 v1

> **位置づけ**: GDD v4 Section 15「オンボーディングとキャリブレーション」の詳細仕様書。本ドキュメントが正として扱い、GDD v4 Section 15はサマリーとして残す。
>
> **参照ドキュメント**: game-design-v4.md / onboarding_assessment_tools_report.md / pmf_research_report.md
>
> **エビデンス基盤**: VB-MAPP, Vineland-3, BRIEF-2, Sensory Profile 2, ABLLS-R の保護者回答可能項目を保護者言語に変換して活用
>
> **設計の核心思想**:
> - 訴求軸は「認知機能向上」ではなく「日常の困りごとの改善（療育サポート）」
> - TTV（価値実感時間）5分以内を死守
> - 認知負荷・入力ストレスの極限排除
> - 困りごとのベースライン数値を初回に取得し、継続後の変化を可視化する素材にする
> - データ取得と体験品質のトレードオフを「質問の導線への自然な埋め込み」で解消

---

## 全体フロー概要

```
【Phase 1: 子どもの基本情報】（約1分・4画面）
  年齢 / 名前 / 発話レベル / 聴覚過敏
      ↓
【Phase 2: 困りごとアセスメント】（約1分・2画面）
  保護者の困りごとカテゴリ + ベースライン数値（頻度）を取得
      ↓
【Phase 3: 子ども向けキャリブレーション】（約3分）
  ゲーム体験として実施 = Tier判定 + 認知プロファイルシードデータ自動取得
      ↓
【Phase 4: アカウント作成】（任意・1分）
  スキップ可。匿名セッションとして継続可能
      ↓
【Phase 5: ホーム到着 + 期待値設定】
  困りごとと連動したコンテンツ表示 + 継続への期待感設定
      ↓
【Phase 6: 詳細アセスメント】（後置・任意・3段階）
  保護者ダッシュボードから段階的に収集
```

**TTV目標**: Phase 1〜3完了まで5分以内（必須画面6画面 + キャリブレーション3試行）

---

## Phase 1: 子どもの基本情報（4画面・約1分）

### Screen 1: 年齢と名前

**目的**: 年齢グループの自動算出（コンテンツ出し分けの基礎変数）

**設問**:
- 年齢（「〇歳」の直接選択ボタン。3〜18歳。日付ピッカーは使わない）
- 名前またはニックネーム（任意。「あとで決める」ボタンを用意）

---

### Screen 2: 発話レベル（コア設問）

**目的**: Tier判定・指示レベル（L1〜L4）・全ゲームのDDA初期値の最重要変数

**元ツール**: VB-MAPP Mand/Tact L1-L3 / Vineland-3 Expressive / ABLLS-R F,G

**設問**: 「お子さまの普段の話し方に一番近いものを選んでください」

| アイコン | 選択肢テキスト | 内部マッピング |
|--------|-------------|-------------|
| 🔇 | まだ言葉は出ていない | 非言語 → Tier 1確定候補 / L1指示 |
| 💬 | 単語をいくつか話す（「ジュース」「ママ」など） | 単語 → Tier 1-2 / L2指示 |
| 💬💬 | 2語で話す（「ジュースちょうだい」など） | 2語文 → Tier 2 / L3指示 |
| 🗣️ | 短い文で話す（「おなかすいた」「あれとって」） | 短文 → Tier 2-3 / L3指示 |
| 🗣️🗣️ | 会話ができる（質問に答えたり話しかけたりする） | 会話 → Tier 3 / L4指示 |

**設計注意点**:
- 「迷ったら下の方を選んでください」と小さく補足（保護者の過大評価バイアス対策）
- アイコンは直感的に。テキストは大きく

---

### Screen 3: タブレット操作能力

**目的**: 全ゲームの前提条件確認。Tier判定の補助変数

**元ツール**: Mullen Fine Motor / ABLLS-R Z: Fine Motor

**設問**: 「お子さまはタブレットや画面を指でタッチして操作できますか？」

| 選択肢 | 内部マッピング |
|--------|-------------|
| 自分でできる | タッチ操作確立 |
| 手を添えてあげればできる | 保護者同伴モード推奨フラグ |
| まだ難しい | Tier 1固定候補 / 保護者同伴必須 |

---

### Screen 4: 聴覚過敏

**目的**: 感覚過敏コントロールUIのデフォルト値を初回から最適化する。ASDの37〜69%が聴覚過敏を持つ（GDD v4 Section 10）

**元ツール**: Sensory Profile 2 Auditory / CARS-2 聴覚反応

**設問**: 「お子さまは、大きな音や突然の音を嫌がりますか？（掃除機・ブザー・花火など）」

| 選択肢 | 感覚過敏UIの初期設定 |
|--------|------------------|
| とても嫌がる | BGM OFF / 効果音 OFF |
| 少し嫌がる | BGM OFF / 効果音 小 |
| 特に気にしない | BGM ON / 効果音 標準 |
| むしろ音が好き | BGM ON / 効果音 大 |

**設計注意点**:
- 「回答すると音の設定を自動で調整します」と説明して回答の意味を伝える
- 「あとで設定画面から変更できます」を補足表示

## Phase 2: 困りごとアセスメント（2画面・約1分）

### Screen 5: 主訴の選択

**目的**:
1. 保護者の自覚的ペインを把握し、保護者ダッシュボードの表示優先度を設定する
2. 「Manasがこの困りごとに対して使えるツールである」という訴求接続を行う
3. 継続後の変化を可視化するカテゴリを決定する

**元ツール**: BRIEF-2 / SCQ / Vineland-3 / CARS-2 / PMFリサーチ保護者ペイントップ項目を統合

**設問**: 「今、お子さまのことで気になっていることを選んでください（最大3つ）」

| アイコン | 選択肢テキスト | 内部タグ | 関連ゲーム |
|--------|-------------|---------|---------|
| 😤 | 気持ちの切り替えが難しい（パニック・かんしゃく） | emotion_regulation | きもちフレンズ / ルールチェンジ |
| 👀 | 集中が続かない、すぐ気が散る | attention | ひかりレスキュー |
| 🗣️ | 言葉でのやりとりが難しい | communication | ことばとえほん / きもちフレンズ |
| 🧩 | 友達や人との関わりが難しい | social | きもちフレンズ |
| 📚 | 読み書きや学習が難しい | learning | ことばとえほん / おぼえてすすむ |
| 👐 | 手先が不器用、身体の使い方が気になる | motor | タッチアドベンチャー |
| 🔄 | 予定や変化に対応するのが難しい | flexibility | ルールチェンジ / たんけんめいろ |
| 💭 | 物事を覚えるのが難しい | memory | おぼえてすすむ |

**設計注意点**:
- 選択後「この困りごとに取り組むコンテンツを準備します」と表示し、Manasとの接続を明示する
- 選択した項目は保護者ダッシュボードの表示優先度に直結する

---

### Screen 6: ベースライン数値の取得

**目的**:
1. 継続後の変化を数値で可視化するためのゼロ点を取得する
2. 実ユーザーデータによる変化グラフの素材にする（実データのため科学的・法的リスクなし）
3. 保護者に「測定が始まった」という実感を持たせる

**表示ロジック**: Screen 5で選択した困りごとのうち最初に選んだものを1つ自動表示

**設問例**（「気持ちの切り替えが難しい」を選んだ場合）:
> 「気持ちの切り替えが難しい場面は、今週何回くらいありましたか？」

| 選択肢 | 内部スコア |
|--------|---------|
| ほぼ毎日（週5〜7回） | 6 |
| 週3〜4回 | 3.5 |
| 週1〜2回 | 1.5 |
| たまに（月数回） | 0.5 |
| 最近はあまりない | 0 |

**設計注意点**:
- 画面タイトルに「これが今日の出発点です」と表示し、変化の可視化への期待感を設定する
- 「4週間後に同じ質問をします。変化が見えてきます」という一言を添える
- 回答をベースラインスコアとして保存。4週後の週次レポートで自動比較する

## Phase 3: 子ども向けキャリブレーション（約3分）

### 設計思想

「テストされている」と感じさせない。「ゲームが始まった」と感じさせる。
保護者からの回答（Phase 1-2）と客観データ（ゲームプレイ）を照合し、Tier判定の精度を高める。

**取得データ**:
- タッチ操作能力（ターゲット精度・微細運動ベースライン）
- 視覚マッチング能力（形・色弁別）
- 抑制能力（Go/No-Go）
- 全ゲームのTier判定シードデータ

---

### キャリブレーションフロー

```
マスコット登場アニメ（10秒）
Lunaが「いっしょにあそぼう！」と声をかける
      ↓
【キャラクター選択】（5秒）
  LunaとMoguraの2体から好きな方を選ぶ
  → 子どもの主体性を確保。最初の「自分で決めた」体験
      ↓
【Trial 1: タッチアドベンチャーLv1】（30秒）
  大きなターゲット（100px以上）をタップするだけ
  → タッチ操作能力・微細運動のベースライン
  → 「すごい！じょうず！」で即時ポジティブフィードバック
      ↓
【Trial 2: くるくるパズルLv1】（30秒）
  同じ形を選ぶだけ（回転なし・2択）
  → 視覚マッチング・色/形弁別のベースライン
  → Tier 2解放の主要判定基準
      ↓
【Trial 3: ひかりレスキューLv4】（45秒）
  Go/No-Go簡易版（標的と非標的の区別）
  → 抑制能力・注意持続のベースライン
  → 発話レベルがTier 2以上の場合のみ実施
      ↓
【結果表示】（20秒）
  「○○ちゃんの得意なこと、わかってきたよ！」
  認知プロファイルのシードデータをバックグラウンドで生成
  Tier判定を確定
```

---

### Tier判定ロジック（Phase 1 + Phase 3の統合）

```
【Step 1: Phase 1の保護者回答から暫定Tier推定】
  発話レベル = 非言語           → 暫定Tier 1
  発話レベル = 単語             → 暫定Tier 1-2
  発話レベル = 2語文以上        → 暫定Tier 2-3
  タブレット操作 = まだ難しい   → Tier 1固定候補

【Step 2: Phase 3キャリブレーション結果で確定・補正】
  Trial 1（タッチ）全試行で無反応またはランダム押し → Tier 1確定
  Trial 2（視覚マッチ）正答率 < 50%                 → Tier 1確定
  Trial 2（視覚マッチ）正答率 ≥ 80%                 → Tier 2解放
  Trial 3（Go/No-Go）成立                            → Tier 3候補
  Trial 3（Go/No-Go）不成立                          → Tier 2確定

【Step 3: 乖離検出と保護者通知】
  Phase 1保護者報告とPhase 3実績が大きく乖離した場合:
  → キャリブレーション結果（客観データ）を優先
  → 保護者ダッシュボードに「実際の難易度を自動調整しました」と通知

【Phase 6での補正（後日入力時）】
  診断「重度ID」→ Tier上限を2に制限
  診断「中度ID」→ Tier3のStop-signal/backwardをロック
  年齢 < 4歳   → Tier3の一部をロック
```

## Phase 4: アカウント作成（任意・約1分）

**方針**: スキップ可能。匿名セッションとして継続できる。

**表示文**: 「○○ちゃんの成長を記録しておきましょう。いつでも続きから始められます。」

**入力項目**: メールアドレス / パスワード

**スキップ時**: 「あとで登録する」ボタンを目立つ位置に配置。データはローカルに一時保存。

## Phase 5: ホーム到着 + 期待値設定

### 冒険マップ表示

Tier判定に応じてアンロックされたゲームが表示される（GDD v4 Section 12参照）。

---

### 期待値設定（初回のみ・スワイプ式2枚）

**目的**: 「続けることで変化が起きる」という継続動機を初回に設計する。困りごととゲームの接続を保護者に伝える。

**1枚目: 困りごととゲームの接続**

Screen 5の選択に連動して文章を自動生成する。

> 例（「気持ちの切り替えが難しい」を選択した場合）:
> 「気持ちの切り替えが難しいと教えてくれましたね。
> Manasでは、気持ちをコントロールする練習ができるゲームを用意しています。
> 一緒に取り組んでいきましょう。」

**2枚目: 変化のタイムライン**

MVP段階（実ユーザーデータなし）:
> 「週3回、続けてみてください。
> 4週間後に、今日と同じ質問をします。
> どんな変化があったか、一緒に確認しましょう。」

ベータ後（実ユーザーデータが取れた段階で更新）:
> 「同じ困りごとを持つお子さまの中で、
> 週3回・4週間続けた場合に変化を感じたという声があります。（n=〇〇名）
> ※個人差があります。すべてのお子さまに同様の変化が現れるわけではありません。」

**アクセス場所**: 保護者ダッシュボード → 「お子さまの情報を追加する」

**誘導タイミング**:

**誘導文**: 「情報を追加するとゲームをより最適化できます（約2分）」

## 保護者ダッシュボード：困りごとの変化の可視化

### ベースライン → 4週後の比較表示

Phase 2 Screen 6で取得したベースライン数値を元に、4週後の週次レポートで自動比較する。

**表示形式（4週後の週次レポート内）**:

```
【4週間の変化】

「気持ちの切り替えが難しい」場面
スタート時:  週5〜7回
今週:        週1〜2回

少しずつ変化が見えてきました 🌱
```

**設計注意点**:
- 数値が変わっていない週も「続けることで見えてくる変化があります」と表示する（挫折防止）
- 改善が見られた場合は強調表示し、保護者の達成感につなげる
- このデータは実ユーザーデータとして蓄積し、ベータ後に期待値設定ページへ匿名表示する
- ベータユーザーから十分なデータが蓄積された段階でOtsimo型の変化グラフとして展開する

### 週次チェックイン（継続負荷の最小化）

毎週日曜のセッション完了直後に自動で表示する。保護者が別途操作する必要はない。

**表示内容（3択・10秒で完了）**:
> 「今週、○○の場面はどのくらいありましたか？」
> → ほぼ毎日 / 週数回 / ほとんどなかった

回答はベースラインとの比較グラフに自動で追加される。

## GDD v4 Section 15との差分サマリー

| 項目 | GDD v4 Section 15（旧） | 本ドキュメント（新） |
|------|----------------------|-----------------|
| Phase 1の画面数 | 3画面（年齢・名前・発話レベル） | 4画面（＋聴覚過敏を追加） |

## 実装優先順

| 順序 | タスク | 理由 |
|------|--------|------|
| 1 | Phase 1（4画面）の実装 | 発話レベルと聴覚過敏はDDA・感覚UIの初期値に直結 |
| 2 | Phase 2（困りごとアセスメント2画面）の実装 | ベースライン取得がないと変化の可視化が機能しない |
| 3 | Phase 3（キャリブレーション）へのキャラクター選択追加 | GDD v4からの変更点はここのみ |
| 4 | Phase 5（期待値設定2枚）の実装 | 困りごととゲームの接続表示。継続動機の核心 |
| 5 | ベースライン → 4週後比較の保護者ダッシュボード実装 | 変化の可視化。継続率に直結 |
| 6 | 週次チェックイン（セッション完了後の自動表示）実装 | 保護者の記録負荷ゼロ化 |
| 7 | Phase 6-P1（3回後の詳細アセスメント）実装 | DDA精密化 |
| 8 | Phase 6-P2 / P3の実装 | 社会性・診断情報の後置収集 |
